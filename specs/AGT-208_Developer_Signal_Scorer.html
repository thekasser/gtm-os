AGT-208 — Developer Signal Scorer

Layer 2 · ICP & lead management · Pre-MQL developer-funnel scoring · Daily batch + event-triggered · Self-serve → sales-handoff progression
L2 · Service 08 Specced · v38 Tier 1 · Service RevOps + Developer Relations
Why this service exists. Developer-led / API-platform businesses have a funnel shape that AGT-201 (ICP Scorer) and AGT-202 (Lead Router) were not designed for: customers don't enter through marketing forms or outbound — they enter as individual developers who sign up, get an API key, and start consuming. By the time a "lead" exists in the traditional sense, the customer has already self-served through significant adoption. AGT-208 scores developers and developer-aggregated accounts on enterprise-progression signal, surfaces sales-handoff candidates *before* they manually request a call (often too late), and never collapses the PLG funnel into the post-MQL world AGT-201 owns.
Purpose

AGT-208 ingests developer-level events from the product (signups, API calls, model selections, deployment changes, documentation engagement, billing inquiries) and produces two scores: a per-developer enterprise-progression score and an account-aggregated score that rolls multiple developers in the same organization into a unified signal. Output drives sales-handoff routing — identifying which self-serve accounts should warm-route to an AE for proactive outreach, and which should stay self-serve. AGT-208 is the upstream complement to AGT-201; once an account crosses the sales-handoff threshold, AGT-201's firmographic ICP score takes over for traditional segment-band routing.

The hard line. AGT-208 never overrides an AGT-201 firmographic ICP determination. If AGT-201 has scored an account as T3 (poor fit) but developer signal is hot, AGT-208 surfaces the contradiction to RevOps for review — it does not auto-promote. The two services run on different data and the disagreement itself is signal.
Per-developer signal dimensions (100 pts base)

Five dimensions, additive. Each capped per spec to prevent any single signal from dominating.

DimensionPts maxWhat it captures
1. Consumption velocity30Volume of API calls / units consumed in trailing 30 days, with growth rate over trailing 90 days. Pattern-recognized: rapid-growth velocity scores higher than flat-volume velocity, even at same absolute level. Reads UsageMeteringLog filtered by developer-API-key.
2. Production signal25Indicators of moving from prototype to production: deployments to dedicated infrastructure (vs serverless), error-rate sensitivity (production-grade SLA queries), SLA / latency percentile API queries, region-pinning, idempotency-key usage.
3. Enterprise context20Corporate email domain (vs gmail / personal), domain seniority (multi-developer presence at same domain), funded-company signals from enrichment, industry-vertical match. Lifted to org level when ≥ 3 developers from same domain are scored simultaneously.
4. Commercial intent15Pricing-page visits, billing dashboard traffic, contract / commit-tier inquiry, security-and-compliance documentation requests (SOC 2, HIPAA, data-residency), procurement-related help-desk tickets.
5. Stakeholder breadth10Number of distinct contacts at the same domain interacting with the product or docs in trailing 30 days. Single developer = low signal; 5+ developers + 1 manager-level inquiry = high signal indicating organizational adoption depth.

Score → tier: 80–100 = handoff-priority · 60–79 = handoff-warm · 40–59 = monitor · < 40 = stay-self-serve.

Domain aggregation — how developer signals roll into account scores

An organization may have 10 developers using the platform, each at modest individual signal but cumulatively significant. Domain aggregation prevents this from being missed.

Aggregation ruleWhy
Per-corporate-domain rollup nightly. Sum of per-developer scores capped at 100 (single account can't double-count). Highest individual score retained as a separate field.Captures collective adoption depth without inflating beyond the individual ceiling.
Domain enrichment via existing AGT-204 Lead Enrichment pipeline. Fallback to email-domain matching with public-domain (gmail/yahoo/outlook) exclusion list.Reuse existing enrichment; don't duplicate.
Account-aggregated tier override: if 3+ developers at same domain each score ≥ 60 (handoff-warm individually), domain-level tier is auto-elevated to handoff-priority regardless of summed score.Multi-developer adoption breadth is itself the signal — bypasses the additive-cap edge case where 5 medium-score devs would otherwise sum to a flat ceiling.
Cross-reference with existing Accounts table by domain. If account exists, attach signal to the account; if not, queue for AGT-201 + AGT-204 enrichment review.Doesn't auto-create accounts — surfaces, for AGT-201 + AGT-204 to ingest.
